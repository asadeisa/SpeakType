## Task 2 — LOCATE error-masking + failure-path defects (the safety net)

### Finding 2.1: Quota check silent stub fallback masks 401/unauthorized & network errors, enabling broken record loop [BLOCKER]

- **Location:** `apps/extension/src/services/api.ts:108-119`
- **Symptom:** A logged-out, unauthorized, or completely offline user can click the mic button, start recording, and record audio, but upon stopping, the upload inevitably fails with a generic error.
- **Root cause:**
  1. Before recording, `content.ts` calls `quotaHelper.checkQuota()`, which requests quota from the backend via `getQuotaViaBackground()`.
  2. Because `api.getQuota()` has a silent catch-fallback stub, the real 401 Unauthorized or network error returned by the backend is caught and swallowed, and it returns a fake positive quota (`{ remainingSeconds: 1680 }`).
  3. Thus, `checkQuota` falsely thinks the user is authenticated and has quota, allowing them to record.
  4. Once they finish recording, `transcribeViaBackground` is called, which hits the real `/api/v1/audio/transcribe` endpoint with no token (or an expired token). This returns a 401, which propagates and results in a generic `'Upload failed. Tap to retry.'` error.
- **Suggested fix:** Remove the silent mock fallback in `api.getQuota()`, allowing auth and network errors to propagate cleanly. In `content.ts` / `useQuota`, handle the propagated error (such as a 401) by displaying a sign-in prompt (e.g. `'Please sign in to use SpeakType'`) and prevent recording from starting.
- **Confidence:** high

### Finding 2.2: Focus/field shift in Error state hijacks recordings and inserts into old inputs [BLOCKER]

- **Location:** `apps/extension/src/entrypoints/content.ts:74-78, 205-209`
- **Symptom:** If an upload fails and the user clicks another text input, the mic button is stuck in error state showing `'Upload failed. Tap to retry.'`. Clicking it uploads the previous field's audio and inserts the result into the OLD field.
- **Root cause:**
  1. `_retainedBlob` and `_targetField` are module-level global variables in `content.ts` that are not cleared or updated on focus shift/unmounting.
  2. When the user shifts focus to a new editable field, `onFocusIn` is triggered. It calls `detachMic()` (unmounting the old Vue app) and `mountMic()` (mounting a new one), but the shared global Pinia `recorderStore` remains in the `'error'` state.
  3. Clicking the new mic pill runs `runToggle`. Since `recorderStore.currentState === 'error' && _retainedBlob` is `true`, it triggers a retry with the old `_retainedBlob`, uploading it and calling `insertTextAtCaret(_targetField, transcript)` targeting the old field.
- **Suggested fix:** Reset the `recorderStore` to `'idle'` and clear `_retainedBlob` / `_retainedDuration` when the mic is detached, or bind these states to the specific field/mount context rather than global module variables.
- **Confidence:** high

### Finding 2.3: Focus/field shift during confirmation leaks preview cards and inserts into old inputs [BLOCKER]

- **Location:** `apps/extension/src/entrypoints/content.ts:101-131`
- **Symptom:** If the confirmation card is open and the user clicks another input, the old preview card immediately appears on the new input, and accepting it inserts text into the old input.
- **Root cause:**
  1. `_showPreview` is a module-level ref and is not reset on detachment.
  2. When `onFocusIn` is called on a new field, it calls `detachMic()` which unmounts the old Vue app, but the promise `_resolveConfirm` is left hanging, and `_pendingTargetField` still points to the old input.
  3. When the new mic is mounted, because `_showPreview` is a global ref, it immediately renders the inline preview card with the old transcript over the new input. Accepting it inserts the text into `_pendingTargetField` (the old field).
- **Suggested fix:** Reset `_showPreview` to `false`, reject/resolve the pending confirmation promise, and clear all pending field references when detaching/unmounting the mic.
- **Confidence:** high

## Task 3 — LOCATE test + build/verify-gate gaps (the proof)

### Finding 3.1: Un-normalized Blob MIME Type parameter causes Groq to fail with 400 Bad Request [BLOCKER]

- **Location:** `apps/backend/server/services/stt.ts:46`
- **Symptom:** The live voice call returns `502 STT_UNAVAILABLE` (and logs "Groq transcription failed: Groq API returned status 400") even though the Groq API key is valid and GET `/models` works.
- **Root cause:** In `stt.ts`, `audio.mime` contains browser-specific codec parameters (e.g., `audio/webm;codecs=opus`). While `stt.ts` extracts `baseMime` for validation, it still passes the full un-normalized `audio.mime` to the `Blob` constructor:

  ```typescript
  const blob = new Blob([audio.bytes as BlobPart], { type: audio.mime });
  ```

  This results in the `file` multipart payload attachment being sent to Groq with `Content-Type: audio/webm;codecs=opus`. Groq (using standard Whisper under the hood) does not recognize the `codecs=opus` parameter, fails to determine the audio container format, and rejects the upload with a `400 Bad Request`.

- **Suggested fix:** Pass the normalized `baseMime` to the `Blob` constructor so it is uploaded with a standard, bare content type (e.g., `audio/webm`):

  ```typescript
  const blob = new Blob([audio.bytes as BlobPart], { type: baseMime });
  ```

- **Confidence:** high

### Finding 3.2: Node.js/undici Buffer-to-Blob Stream serialization failure in Multipart body [BLOCKER]

- **Location:** `apps/backend/server/services/stt.ts:46`
- **Symptom:** Groq returns a `400 Bad Request` (Invalid audio file), empty/zero-byte payload, or a `502 STT_UNAVAILABLE` server error because the uploaded file data is corrupted or truncated.
- **Root cause:** In modern Node.js runtimes, `globalThis.fetch` is backed by `undici`. When a `Blob` is constructed directly from a Node.js `Buffer` object (e.g., `new Blob([audio.bytes as BlobPart])`), the multipart form-data serializer in `undici` can fail to correctly serialize the buffer stream, or it can serialize the entire shared Node memory pool (due to pooling of `Buffer.allocUnsafe` or standard Buffer allocation), causing data corruption or a zero-byte body transmission.
- **Suggested fix:** Convert the Node `Buffer` to a clean, un-pooled, standard `Uint8Array` before passing it to the `Blob` constructor to guarantee correct stream serialization:

  ```typescript
  const blob = new Blob([new Uint8Array(audio.bytes)], { type: baseMime });
  ```

- **Confidence:** high
